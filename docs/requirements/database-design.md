# Database Design

## Purpose

The Oracle database is the persistent store for the Online Hospital Management System. The initial schema supports identities and roles, clinical organization, patients, appointments, medical records, prescriptions, billing, payments, and audit history. Business workflows will be implemented in later phases through JDBC DAO and service classes.

## Tables and responsibilities

| Table | Responsibility |
| --- | --- |
| `ROLES` | Supported authorization roles. |
| `USERS` | Login identities, account state, and password hashes. |
| `DEPARTMENTS` | Hospital departments and locations. |
| `DOCTORS` | Doctor profiles linked to users and departments. |
| `PATIENTS` | Patient identity, demographic, and contact data. |
| `DOCTOR_SCHEDULES` | Recurring weekly availability and slot durations. |
| `APPOINTMENTS` | Bookings between patients and doctors. |
| `MEDICAL_RECORDS` | Clinical findings and treatment notes for visits. |
| `PRESCRIPTIONS` | Prescription headers linked to clinical encounters. |
| `PRESCRIPTION_ITEMS` | Medicines and directions on each prescription. |
| `BILLS` | Bill totals, balances, and settlement state. |
| `BILL_ITEMS` | Individual consultation, room, medicine, test, and procedure charges. |
| `PAYMENTS` | Payment transactions received against bills. |
| `AUDIT_LOGS` | Security-sensitive and business-event history. |

## Primary-key strategy

Every table has a numeric surrogate primary key defined as `NUMBER GENERATED BY DEFAULT AS IDENTITY`. Oracle generates normal keys while `BY DEFAULT` still permits controlled data migration when an explicit identifier is necessary. All primary-key constraints are named.

## Foreign-key relationships

- Users belong to roles.
- Doctors belong to departments and have a one-to-one user identity.
- Patients may have a one-to-one user identity; a portal account is optional.
- Doctor schedules belong to doctors.
- Appointments connect patients and doctors and record the user that created the booking.
- Medical records connect patients and doctors and may reference an appointment.
- Prescriptions connect patients and doctors and may reference an appointment and medical record.
- Prescription items belong to prescriptions.
- Bills belong to patients and may reference appointments; bill items belong to bills.
- Payments belong to bills and reference the receiving user.
- Audit events may reference the responsible user; the relationship is nullable for anonymous or system actions.

Tables are created in parent-before-child order. The cleanup script drops them in reverse dependency order.

## Important constraints

Named unique constraints protect role names, usernames, user emails, department names, doctor registration numbers, patient numbers, bill numbers, and payment numbers. Composite unique constraints prevent duplicate doctor schedule slots and duplicate doctor appointment bookings.

Named check constraints restrict all status and category columns to documented values. They also enforce non-negative monetary amounts, positive quantities and payment amounts, positive schedule durations, valid `IS_AVAILABLE` values (`0` or `1`), and `HH24:MI` schedule/appointment time formats. Start and end times use fixed-width `VARCHAR2(5)` values so JDBC can handle recurring times without applying an unintended date or time-zone conversion.

## Index strategy

Foreign-key and frequent-filter columns have explicit indexes where requested: user role, doctor department, patient phone/email, appointment patient/doctor/date, medical-record patient, bill patient/status, payment bill, and audit user/creation time. Oracle automatically creates indexes for primary and unique constraints; duplicate explicit indexes are therefore avoided.

## Password storage policy

`USERS.PASSWORD_HASH` stores only a strong, salted, one-way password hash produced by the application. Plaintext passwords, reversible encrypted passwords, default user passwords, and application-user seed records are prohibited. Database credentials reside only in the ignored local `db.properties` file.

## HOSPITAL_APP schema user

`HOSPITAL_APP` owns and accesses the application objects. The setup script grants only session and schema-object creation capabilities plus a limited quota on `USERS`. It does not grant `DBA` or broad administrative privileges. Its placeholder password must be replaced locally before the script is executed.

Oracle `SYSTEM` must not be used as the runtime account. `SYSTEM` is an administrative identity with access far beyond the application. Using a least-privilege schema limits accidental damage and reduces the impact of a compromised application connection.

## Oracle-specific data types

- `NUMBER` is used for keys, counts, flags, measurements, and fixed-scale monetary values.
- `VARCHAR2` stores bounded text and fixed-format recurring times.
- `DATE` stores calendar dates where fractional seconds are unnecessary.
- `TIMESTAMP DEFAULT CURRENT_TIMESTAMP` records event and audit times.
- `CLOB` stores potentially long clinical notes and audit before/after values.
- Oracle has no SQL `BOOLEAN` table-column type in this target design; `NUMBER(1)` with a check constraint represents boolean state.

## JDBC connection pooling

`DatabaseConnectionManager` lazily creates one HikariCP pool for the web application from classpath `db.properties`. It validates all mandatory properties, never logs the password, returns standard JDBC connections, and closes the pool through `ApplicationLifecycleListener` when Tomcat undeploys the application. Lazy initialization prevents temporary database downtime from blocking web-application deployment.

## Audit logging approach

`AUDIT_LOGS` records the actor, action, entity name and identifier, optional old/new representations, originating IP address, user agent, and timestamp. Application services will write audit events for security-sensitive and important business changes. Audit records should be append-only to normal application users; sensitive values such as passwords and password hashes must never be copied into audit fields.

## SQL execution order

1. Connect to `XEPDB1` as an administrative user and run `database/create-app-user.sql` after replacing its password placeholder.
2. Reconnect as `HOSPITAL_APP`.
3. For a reset, run `database/cleanup.sql`.
4. Run `database/schema.sql`.
5. Run `database/seed-data.sql`.

The scripts are intentionally not executed by the Maven build. Database deployment remains an explicit administrative step.
